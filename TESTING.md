# Testing Guide

This repository is a .NET 10, event-driven reconnaissance pipeline with a Blazor/Minimal API Command Center, Gatekeeper admission flow, worker services, EF Core/Postgres persistence, Redis dedupe, and MassTransit/RabbitMQ messaging.

## Current testing state

Before this update, the repository had two xUnit projects:

- `NightmareV2.Workers.Enum.Tests`
  - subdomain normalization/parsing
  - enum consumer behavior with deterministic fakes/mocks
  - wildcard DNS detection
- `NightmareV2.TechnologyIdentification.Tests`
  - technology catalog loading
  - pattern parsing
  - scanner behavior
  - cookie and HTML signal extraction

The largest gaps were gatekeeper orchestration, canonicalization/scope rules, URL/soft-404 classification, high-value regex matching, spider link/API extraction, Command Center input normalization, coverage collection, and repeatable test commands.

## Test strategy

### Unit tests

Use unit tests for deterministic logic that has no external process, database, broker, network, or clock dependency.

Covered unit areas now include:

- `NightmareV2.Application.Tests`
  - asset category and graph relationship rules
  - `UrlFetchClassifier` soft-404 detection
  - `GatekeeperOrchestrator` fan-out, dedupe, scope, failure release, and port-scan enqueue behavior
  - high-value regex matcher behavior
  - high-value pattern and wordlist catalog loading
- `NightmareV2.Infrastructure.Tests`
  - `DefaultAssetCanonicalizer`
  - `DnsTargetScopeEvaluator`
  - `DefaultHttpRequestQueueStateMachine`
- `NightmareV2.Workers.Spider.Tests`
  - API endpoint inference
  - query parameter extraction
  - bounded HTTP body reading
  - HTML/Markdown/JavaScript/plain-text link harvesting
  - adaptive concurrency decisions
- `NightmareV2.CommandCenter.Tests`
  - target root normalization and bulk-import line splitting
- Existing technology and enum worker test projects remain in place and now participate in coverage collection.

### Integration tests

The current integration-style tests are in-process and deterministic. They exercise service/module boundaries with fakes rather than requiring Postgres, Redis, RabbitMQ, Docker, or external binaries.

Keep this approach for fast pull-request checks. Add true external integration tests later for:

- EF Core persistence against Postgres
- Redis dedupe semantics
- MassTransit/RabbitMQ outbox and inbox behavior
- HTTP queue lease/retry concurrency behavior
- enum providers using fake `subfinder`/`amass` binaries

When those are introduced, isolate them under a dedicated test project or trait so they are not required for the fast unit loop.

### Smoke tests

Smoke tests verify that a deployed or locally running stack is alive and can serve essential endpoints.

Use:

```bash
./scripts/test.sh smoke
```

This delegates to the existing `deploy/smoke-test.sh`, which checks:

- `/health`
- `/health/ready`
- Blazor static assets
- diagnostics endpoints

Set `BASE_URL` when testing a non-default host:

```bash
BASE_URL=http://localhost:8080 ./scripts/test.sh smoke
```

### End-to-end tests

The E2E test intentionally covers only the highest-value operator workflow:

1. Command Center is reachable.
2. A target can be created through `POST /api/targets`.
3. The target appears in `GET /api/targets`.
4. The test cleans up with `DELETE /api/targets/{id}`.

Run it against a running local stack:

```bash
BASE_URL=http://localhost:8080 ./scripts/test.sh e2e
```

The E2E script is:

```text
tests/e2e/command-center-target-workflow.sh
```

It uses `curl` and Python's standard `json` module. It does not require `jq`.

## Naming conventions

Use the pattern:

```text
<ClassOrFeatureUnderTest>Tests.cs
```

Name test methods as:

```text
MethodOrFeature_Condition_ExpectedOutcome
```

Examples:

```csharp
LooksLikeSoft404_DetectsStructuredJson404Signals
ProcessAsync_OutOfScope_ReleasesReservedDedupeKey
TryInferEndpoint_TemplatesNumericAndUuidSegmentsAndNormalizesAuthority
```

## How to run tests

### Full fast suite

```bash
./scripts/test.sh
# or
./scripts/test.sh full
```

### Unit tests

```bash
./scripts/test.sh unit
```

### Integration-style tests

```bash
./scripts/test.sh integration
```

### Coverage

```bash
./scripts/test.sh coverage
```

Coverage output is written under `TestResults/` as Cobertura XML files generated by `coverlet.collector`.

You can also run the solution directly:

```bash
dotnet test NightmareV2.slnx --collect:"XPlat Code Coverage"
```

### Individual project

```bash
dotnet test tests/NightmareV2.Application.Tests/NightmareV2.Application.Tests.csproj
dotnet test tests/NightmareV2.Workers.Spider.Tests/NightmareV2.Workers.Spider.Tests.csproj
```

### CI

GitHub Actions configuration was added at:

```text
.github/workflows/tests.yml
```

It restores, builds, and runs the full test suite with coverage on pushes and pull requests.

## Fixtures, fakes, and mocks

Prefer hand-written fakes for application orchestration tests when the behavior is simple and important to assert.

Current examples:

- `GatekeeperOrchestratorTests`
  - fake deduplicator
  - fake scope evaluator
  - fake persistence
  - fake outbox
  - fake worker-toggle reader
- existing enum tests
  - fake enumeration providers
  - capturing outbox
  - Moq for MassTransit consume contexts and simple service dependencies

Use Moq only when implementing a fake would add noise without improving assertions.

## External dependencies

Default test runs should not call:

- live DNS
- internet hosts
- Docker
- Postgres
- Redis
- RabbitMQ
- external command-line tools such as `subfinder` or `amass`

When external dependencies are needed:

1. Keep them out of the default `unit`/`full` fast path.
2. Use seeded fixtures and deterministic cleanup.
3. Mock or fake time, randomness, file system, network, secrets, and process execution where possible.
4. Document required environment variables in this file.

## Coverage expectations

Prioritize meaningful coverage over a fixed percentage.

Expected high coverage areas:

- business rules
- normalization/canonicalization
- state transitions
- worker decision logic
- parsers and classifiers
- failure/negative paths

Lower coverage is acceptable for:

- `Program.cs` composition
- Docker/AWS deploy shell orchestration
- EF Core model boilerplate
- UI markup
- code that requires real infrastructure until Testcontainers-style tests are added

## Maintenance process

For every behavior change:

1. Add or update a test that fails without the change.
2. Keep tests deterministic and independent.
3. Prefer assertions on observable outcomes over implementation details.
4. Add fixtures/builders when setup is repeated in three or more tests.
5. Run `./scripts/test.sh unit` before committing.
6. Run `./scripts/test.sh coverage` before larger merges.
7. Run smoke/E2E tests before deployment or after infrastructure changes.

## Known gaps and next steps

Recommended next additions:

- Postgres-backed integration tests for:
  - target creation uniqueness
  - asset persistence and relationship validation
  - outbox enqueue/dispatch state transitions
  - HTTP queue leasing/retry expiration
- Redis integration tests for dedupe reserve/release/clear behavior.
- MassTransit test harness tests for inbox dedupe and consumer idempotency.
- Command Center route tests using a minimal test host once startup dependencies are made test-host friendly.
- Shell-level deploy tests for `deploy/*.sh` using dry-run/fake docker commands.
- Browser-level Blazor UI tests only for the most important operator flows.
