In-depth design overview

Argus Engine is an event-driven reconnaissance pipeline.

The platform separates intake, orchestration, storage, and execution so each part can be reasoned about, tested, scaled, and operated independently.

System responsibilities

Clear boundaries make the platform easier to explain and safer to operate.

Control plane

The Command Center owns operator interaction: target import, queue visibility, diagnostics, runtime status, and reliability indicators. Public copy should describe this as an operations surface, not a consumer SaaS dashboard.

Admission plane

Gatekeeper decides whether work should enter the pipeline. It is the right place to emphasize normalization, scope evaluation, deduplication, failure release, and controlled fan-out.

Execution plane

Workers perform bounded tasks from queues. This keeps enumeration, spidering, scanning, technology identification, and signal extraction independently deployable.

Persistence and messaging

Postgres stores durable state, Redis supports deduplication/coordination, and RabbitMQ carries MassTransit events between services. Do not claim “agentless,” “serverless,” or “single binary” behavior.

Event flow

How a target becomes a review queue.

1. Operator submits approved scope

A root domain, host, or environment is imported through the Command Center or configured ingestion path.

2. Gatekeeper normalizes and checks

The system canonicalizes target roots, evaluates scope boundaries, and suppresses duplicate work before publishing events.

3. Workers expand observations

Specialized workers discover subdomains, reachable endpoints, service hints, technologies, links, API shapes, and high-value patterns.

4. State is persisted and surfaced

Assets, relationships, queue metrics, reliability signals, and findings are stored for review in the Command Center.

5. Humans validate

Researchers reproduce, assess impact, remove false positives, and decide what is reportable.

Worker roles

Describe workers by evidence collected, not hype.

Worker Primary role Suggested copy
Enumeration Subdomain and target expansion. “Discovers candidate assets and emits normalized observations for downstream checks.”
Port Scan Service exposure discovery. “Identifies reachable services for approved hosts without implying exploitability.”
Spider HTTP content traversal and link extraction. “Harvests links, parameters, and API hints from fetched content for review.”
Technology Identification Fingerprinting from HTML, headers, cookies, and catalogs. “Turns response signals into technology labels that guide manual testing.”
High-Value Pattern matching for noteworthy paths or indicators. “Prioritizes leads that deserve human validation.”

Data model language

Use asset graph terms.

Use consistent nouns: target roots are approved starting points, assets are normalized discovered entities, observations are collected signals, relationships connect assets, queue state describes work progress, and findings are review items rather than confirmed vulnerabilities.

This language avoids overclaiming. It also maps cleanly to the tested areas called out in the repository: asset category rules, graph relationship rules, scope evaluation, URL classification, queue state transitions, and worker behavior.
