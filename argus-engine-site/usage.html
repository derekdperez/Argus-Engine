Usage tutorial

Operate Argus Engine as a recon workflow, not a push-button attack tool.

The recommended workflow is deliberate: confirm authorization, import scope, let workers collect evidence, and manually validate every lead before reporting.

Before every run

Build an engagement record.

Tutorial flow

From approved target to reviewed findings.

Step 1 — Start the local stack

Use the compose-based setup from the repository so the Command Center, workers, Postgres, Redis, and RabbitMQ start together.

docker compose -f deploy/docker-compose.yml up -d --build
Step 2 — Open the Command Center

Use the local URL exposed by the running web service. Confirm that queues, worker activity, and reliability views are healthy before importing scope.

Step 3 — Import authorized target roots

Add only approved domains, hosts, or lab environments. Keep notes about the source of authorization and any limits that should affect worker configuration.

Step 4 — Let Gatekeeper admit work

Gatekeeper should normalize roots, apply scope rules, deduplicate work, and fan out events. If work is rejected, fix the scope record instead of bypassing controls.

Step 5 — Review observations

Use the Command Center to inspect discovered assets, service hints, technology labels, spidered URLs, high-value indicators, and queue state.

Step 6 — Validate manually

Argus should not be described as confirming vulnerabilities by itself. A researcher must reproduce behavior, assess impact, and remove false positives.

Tone guidance

Replace “guaranteed bugs” with “better triage.”

Recommended phrasing: “Argus Engine reduces repetitive recon work and surfaces leads for review.” Avoid claiming automatic exploitation, guaranteed vulnerabilities, stealth scanning, bypassing rate limits, or hands-off reporting.
