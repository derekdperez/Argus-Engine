Architecture

Built around queues, boundaries, and observable state.

The core shape is intentionally familiar to enterprise .NET systems: clear modules, deterministic business rules, isolated workers, and infrastructure kept behind interfaces so testing stays fast and meaningful.

01 / Intake

Targets enter through Command Center

Normalize input, split bulk submissions, validate target roots, and create operator-visible work.

→
02 / Admission

Gatekeeper protects the pipeline

Evaluate scope, reserve dedupe keys, release failures, and enqueue only approved work.

→
03 / Execution

Workers process durable queues

Enumeration, spidering, technology detection, URL classification, and queue state transitions.

Testing philosophy

Fast deterministic tests before heavyweight infrastructure tests

The current test strategy favors unit and in-process integration-style tests using fakes/mocks where useful, keeping the core logic independent from live DNS, Docker, Postgres, Redis, RabbitMQ, or external binaries in the default test loop.

Tech stack

C# at the center, web technologies around the edges

The repository’s language mix is led by C#, with HTML, JavaScript, Shell, CSS, PowerShell, and Python supporting the application, UI, deployment, and maintenance workflows.
