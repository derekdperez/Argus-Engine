Setup instructions

Run Argus Engine with the repository’s actual stack.

The current repo is a .NET 10 solution with Docker-based local orchestration. These instructions avoid imaginary install commands and stay close to the files that exist today.

Prerequisites

Install the platform tools first.

Git

Required to clone the repository and manage local changes to configuration, docs, and deployment files.

Docker + Compose

Recommended for a full local stack because the compose file wires together the web service, workers, Postgres, Redis, and RabbitMQ.

.NET 10 SDK

Required for source builds, tests, and direct project execution outside containers. Keep SDK and package versions aligned with the repo.

Path A

Run the local Docker stack.

Use this path when you want the closest approximation of the platform topology.

Local stack 
git clone https://github.com/derekdperez/Argus-Engine.git
cd Argus-Engine

# Build and start the full local stack.
docker compose -f deploy/docker-compose.yml up -d --build

# Confirm services are running.
docker compose -f deploy/docker-compose.yml ps

# Follow logs while the stack settles.
docker compose -f deploy/docker-compose.yml logs -f

Path B

Build and test from source.

Use this path when you are editing code, validating project references, or running deterministic test suites.

Source workflow 
git clone https://github.com/derekdperez/Argus-Engine.git
cd Argus-Engine

dotnet --info
dotnet restore NightmareV2.slnx
dotnet test NightmareV2.slnx

# Run the web/Command Center project during development.
dotnet run --project src/NightmareV2.CommandCenter/NightmareV2.CommandCenter.csproj

Configuration checklist

Do this before importing any real target.

Connection strings

Confirm Postgres, file-store database, Redis, and RabbitMQ settings. The compose defaults are for local development, not production secrets.

Worker scale

Start conservatively. Increase worker replicas only after you understand queue depth, external tool behavior, and program rate limits.

Scope policy

Document allowed roots, exclusions, windows, and rate expectations before the Gatekeeper admits work.

Diagnostics exposure

Do not expose diagnostics or maintenance endpoints on the public internet without authentication, rate limits, and audit logging.

Troubleshooting

Common setup checks.

Containers build but services do not connect: verify the compose network, service names, and connection strings.

Database startup fails: reset local volumes only if you can lose local data, then rebuild.

Workers are idle: confirm RabbitMQ is reachable, Gatekeeper is publishing events, and the relevant worker service is running.

Tests fail after dependency changes: restore packages from a clean workspace and avoid upgrading package versions without checking project compatibility.
